Proceso de decisión de ACL estándar

Las ACL estándar solo examinan la dirección IPv4 de origen. El destino del paquete y los puertos involucrados no se tienen en cuenta.

El proceso de decisión de una ACL estándar se detalla en la ilustración. El software IOS de Cisco prueba las direcciones en relación con cada una de las condiciones de la ACL. La primera coincidencia determina si el software acepta o rechaza la dirección. Dado que el software deja de probar las condiciones después de la primera coincidencia, el orden de las condiciones es fundamental. Si no coincide ninguna condición, la dirección se rechaza.

La figura muestra un diagrama de flujo con el funcionamiento de una ACL estándar. Un paquete entrante ingresa al router y se consulta si hay una lista de acceso estándar en esta interfaz. Si la respuesta es negativa, se consulta si la dirección IP de destino está en la tabla de routing. Los resultados de esta consulta se analizan más adelante. Si hay una ACL estándar en la interfaz, se consulta si la dirección de origen coincide con una entrada de la lista de ACL. Si la respuesta es negativa, se consulta si ésta era la última entrada de ACL. Si la respuesta es afirmativa, el paquete se envía a la papelera y se envía un mensaje que indica que no se halló el destino. Si la respuesta es negativa, se pasa a la siguiente instrucción para avanzar por las entradas de la lista de ACL. Si la dirección de origen coincide con una entrada de la lista de ACL, se consulta si dicha entrada está en estado de permitir o de denegar. Si está en estado de denegar, el paquete se envía a la papelera y el router envía un mensaje que indica que no se halló el destino. Si la entrada está en estado de permitir, se consulta si la dirección IP de destino está en la tabla de routing. Si la respuesta es negativa, el paquete se envía a la papelera y se envía un mensaje que indica que no se halló el destino. Si la dirección IP de destino está en la tabla de routing, el paquete se envía a la interfaz saliente.
